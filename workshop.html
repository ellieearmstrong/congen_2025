Getting started

Hello! Today we'll be going through some hands-on activities to help you get familiar with the steps involved in genome assembly and quality assessment.

The first thing you should do if you haven't done so is connect to the ConGen server. We'll be working exclusively in the RStudio browser interface that you should be familiar with by now, but if you have questions or problems at any point please feel free to ask! Just in case, here's an annotated picture of what you should roughly be seeing right now. If you are seeing something drastically different or something that you don't understand, let us know.

Most of our work will be done as bash commands typed in the Terminal provided by RStudio. Throughout this walkthrough, commands will be presented as follows:

this is an example command

Under each command, we’ll include a small table that breaks down each part of the command so you can see what every argument is doing:

Command line parameter Description
this Example program being called on the command line.
is Example option used in this demonstration command.
an Another example option used by the example program.
example Yet another illustrative option for the example.
command The final token completing the example command.

The goal of providing these tables is to chip away at the “black box” feeling that command-line tools can sometimes have. If you already feel comfortable with the syntax, you can skim or skip these tables; if not, they’re there to help.

Tip — Help menus

A general convention among command-line software is to provide a help menu listing common options. You can usually view it with:

<program> -h # or <program> <sub-program> -h

For many Linux commands, longer documentation is available with the man (manual) system:

man <command>

The manual viewer can be navigated with the arrow keys and exited with q. If you’re ever unsure what a command or option does, these are great first places to look.

Commands that you should actually run during the workshop will have a green background. Commands that are just for illustration (or that you don’t need to run yourself) will have a red background, like this:

this is an example command that won't be run

At several points, we’ll pause and look at the output of a program together. When we do, we’ll usually show a short snippet of the output so that you can compare what you see on your screen with the walkthrough:

Here is some made up output. Looking at your data is very important! You can catch problems before you use the data in later analyses.
Tip — Looking at your data

A crucial part of bioinformatics is simply looking at your data and intermediate outputs. We’ll pause at several points to inspect what a command produced so that you can catch problems early.

Hornbill data

For this workshop we will focus on hornbills (family Bucerotidae), a clade of large, charismatic frugivores found across the forests and savannas of Africa and Asia. Hornbills are key seed dispersers and cavity nesters, and many species are of significant conservation concern due to habitat loss and hunting.

Gonzalez et al. (2013) generated a comprehensive molecular phylogeny for all extant hornbill species, combining mitochondrial and nuclear loci to resolve relationships across the family. Their results reveal several deep clades, show that some traditional genera are not monophyletic, and highlight strong splits between African and Asian lineages. Many hornbill species, particularly in Southeast Asia, are globally threatened, underscoring the importance of genomic data for conservation planning.

In this module, we will use a subset of hornbill genomic data as a worked example for genome assembly and read mapping. You can think about where your focal individual or species might sit on the hornbill tree, and how genome assemblies ultimately tie into questions about diversification, biogeography, and extinction risk.

Using these data, we will:

  • Assemble a hornbill genome and evaluate basic assembly statistics (contiguity, completeness).
  • Map reads back to the assembly (and to alternative references) and compare mapping and depth profiles.
  • Discuss how assembly and mapping choices influence downstream phylogenetic and conservation inferences for hornbills and other taxa.

Input files & output prep

Copying input data

For this workshop we’ll primarily work with sequence data in FASTA and FASTQ formats. The instructor has placed a set of input files on the server in a shared directory. We’ll start by copying that directory into our home directory so we can work on our own independent copy.

First, move to your home directory:

cd ~
Command line parameter Description
cd The Linux “change directory” command.
~ Shortcut that means “my home directory” for the current user.

Now copy the workshop data directory into your home directory:

cp -r instructor_materials/Gregg_Thomas/congen-assembly/ .
Command line parameter Description
cp The Linux copy command.
-r Copy directories recursively, including all files and subdirectories.
instructor_materials/Gregg_Thomas/congen-assembly/ Path to the directory we’re copying from on the shared server.
. Destination path; here . means “copy it here, keeping the same name”.

Change into the new directory:

cd congen-assembly
Command line parameter Description
cd The change-directory command again.
congen-assembly The name of the folder we just copied; we’ll do most of the workshop inside this directory.

Then check what’s inside:

ls
Command line parameter Description
ls List the contents of the current directory (files and folders).

You should see several subdirectories, for example:

Folder Description
dmel-3R-reference/ Reference sequence and indices for a comparison genome (used later for mapping exercises).
dpse-chr2-reads/ Example short-read data for the assembly and mapping exercises.
expected-outputs/ Pre-computed outputs for each step, in case something fails or takes too long.
scripts/ Helper scripts for some parts of the analyses.
Tip — Pre-generated outputs

If you get stuck on a command or your job is taking too long, you can often continue to the next step by using the corresponding file from expected-outputs/ instead of the file you were supposed to create. When in doubt, ask an instructor which file to use.

Preparing output directories

It’s helpful to plan ahead for where your outputs will go so that your working directory stays organized. For this module we’ll generate assemblies, read mappings, and alignments, so we’ll create a separate directory for each.

mkdir alignments
Command line parameter Description
mkdir Create a new directory (folder).
alignments Directory that will hold alignment files when we compare assemblies and mapped reads.
mkdir assemblies
Command line parameter Description
mkdir Make another new directory.
assemblies Directory where we’ll store genome assemblies generated during the workshop.
mkdir mapped-reads
Command line parameter Description
mkdir Make a third directory.
mapped-reads Directory that will hold BAM or SAM files from read mapping steps.

Some tools require that you create an output directory ahead of time. For example, FastQC expects a directory where it can place its reports:

mkdir fastqc-output
Command line parameter Description
mkdir Create a new folder.
fastqc-output Destination folder for FastQC reports generated during quality checking.

Other programs will create their own subdirectories automatically. At this point you should have a clean structure ready to hold the results of each step in the assembly and mapping workflow.
